## [v2] Нейроответы — стратегия

### Когда коммерческая страница попадает в нейроответ

AI-поисковики (Яндекс Нейро, Google AI Overviews, ChatGPT Search) генерируют ответы **преимущественно на информационные запросы**.

| Тип запроса | Шанс нейроответа | Примеры |
|---|---|---|
| Транзакционный | Очень низкий | «купить iPhone 16 Pro Крым», «цена MacBook Air» |
| Коммерческий с условием | Низкий | «гарантия на iPhone в Maxmobiles», «трейд-ин iPhone Севастополь» |
| Сравнительный / выборочный | **Средний** | «чем iPhone 16 Pro отличается от 15 Pro», «какой MacBook лучше» |
| Информационный про продукт | **Высокий** | «как выбрать iPhone», «что важно в MacBook для учёбы» |
| Информационный про магазин | **Высокий** | «какая гарантия на iPhone в Maxmobiles», «как работает трейд-ин» |

**Коммерческие страницы Maxmobiles цитируются в нейроответах через:**
1. **FAQ-блок** — прямые ответы на информационные вопросы покупателей
2. **`.mm-intro-text`** — самодостаточные абзацы с конкретными фактами
3. **`WebPage.description`** в JSON-LD — первый источник для Яндекс Нейро и ChatGPT Search
4. **Блог-статьи** — основной канал для информационных запросов (вне scope этого скилла)

**Практическое правило:** оптимизация мета-тегов влияет на нейроответы косвенно (через `WebPage.description`). Основная работа по нейроответам — в контенте страницы (shop-page-builder-v2).

### Флаг нейроответа

Если пользователь явно упоминает нейроответы, AI Overviews, Яндекс Нейро или хочет «цитироваться» — активировать **информационный маршрут** в Step 1 и Step 2d.

## Приоритет трафика: Google и Яндекс одинаково важны

Реальные данные аналитики Maxmobiles:
- **Google**: **56%** органического трафика · средний чек на **20% выше**, чем с Яндекса
- **Яндекс**: **43%** органического трафика · количество заказов примерно равно Google
- **Расчётный вес выручки**: Google ≈ **55%**, Яндекс ≈ **45%**

**Принцип:** оптимизируем под Google в первую очередь (больший трафик + крупные покупки), Яндекс — в приоритете наравне (равный объём заказов). Обе системы критически важны — нельзя жертвовать одной ради другой.

**Формула эффективного веса (на основе доли выручки):**
```
Эффективный вес = (Google-значимость × 0.55) + (Яндекс-значимость × 0.45)
```

## Workflow (execute in order)

### Step 1 — Parse the request

Extract from the user's message:
- **Page type**: конкретная модель / серия / топовая категория / аксессуары / сервис-услуга
  - **Исключение — пагинация / фильтры** (`?page=2`, `?features_hash=...`): скилл не генерирует мета-теги для этих URL. Решение — `canonical` на первую страницу категории или `noindex` на уровне CS-Cart настроек.
- **Product name / category** (e.g., iPhone 16 Pro Max, MacBook Air M4, AirPods Pro 3, ремонт iPhone)
- **Keyword data**: если пользователь предоставил список ключей с частотностью/значимостью
- **Quantity**: single page or batch
- **[v2] Флаг нейроответа**: упомянуты ли нейроответы, AI Overviews, «цитируемость» — если да, активировать Step 2d
- **[v2] Мультибренд**: если в названии категории, URL или описании пользователя упомянуто 2+ бренда — зафиксировать список **вторичных брендов** (все кроме Apple), активировать **Step 2e**. Apple всегда основной бренд для SERP-запросов.

**Маршрут выполнения:**

```
keyword data НЕ предоставлена → Step 2 (3 запроса) → [Step 2e если мультибренд] → [Step 2d если флаг] → Step 2b-Wordstat → Step 2b → Step 2c → Step 3
keyword data предоставлена    → Step 2-mini (3 запроса) → [Step 2e если мультибренд] → [Step 2d если флаг] → Step 2b-Wordstat → Step 2b → Step 2c → Step 3
```

**Step 2b-Wordstat выполнять всегда**, даже если пользователь уже дал формулировку запроса — она может оказаться не самой частотной в кластере (см. Step 2b-Wordstat). Исключение: если пользователь предоставил **измеренную** региональную частоту (не оценку значимости) по каждому ключу — тогда Step 2b-Wordstat можно пропустить, его данные приоритетнее.

**Step 2c (Gap Analysis) выполняется всегда.** При пропущенном основном Step 2 — выполнить **Step 2-mini**: 2 WebSearch параллельно (обязательные):
1. `"купить [Категория/Модель] Крым"` — поиск реальных конкурентов
2. `"[Категория/Модель] Севастополь"` — региональные конкуренты

**Опциональный запрос №3** — только если планируется HTML-блок (shop-page-builder-v2):
3. `"как выбрать [Категория/Модель]"` — FAQ-семена для shop-page-builder-v2

После Step 2-mini — также выписать домены из топ-10 по каждому запросу и проверить наличие приоритетных конкурентов Maxmobiles (полный список — в Step 2). Найденных пометить **★** для Step 2c.

**Batch-режим — стратегия SERP-запросов:**
- **2–4 страницы**: запустить WebSearch для каждой страницы параллельно (N×3 одновременно).
- **5+ страниц**: группировать похожие категории. SERP по «купить iPhone Крым» переиспользовать для всех серий iPhone. Gap Analysis — один общий, если страницы из одной категории.

Выполнять когда: keyword data не предоставлена.

Кириллические варианты брендов для запроса №3 — см. таблицу в Step 2b.

Run **3 WebSearch queries in parallel** (обязательные):

1. `"купить [Категория/Модель] Севастополь"` — title-паттерны приоритетного города
2. `"купить [Категория/Модель] Крым"` — региональные паттерны конкурентов
3. `"[Кириллический вариант] купить крым"` — реальная частотность кириллики в региональной выдаче

По каждому запросу явно выписать домены из первых 10 результатов (только домен верхнего уровня: `wildberries.ru`, не `www.wildberries.ru`). Это основа для расчёта доминирования агрегаторов в Step 2b.

Агрегаторов невозможно обогнать — изучать их title бессмысленно.

После выписки доменов из топ-10 — проверить, присутствует ли кто-то из известных конкурентов:

| Домен | Тип |

| i-store.net | Apple-магазины, федеральный |

| allohastore.ru | региональный, Севастополь |

**Когда выполнять:** Step 1 зафиксировал 2+ бренда на странице. Выполнять параллельно со Step 2d (или сразу после Step 2, если Step 2d не активирован).

**Для каждого вторичного бренда** (не Apple) — запустить **2 WebSearch параллельно**:

1. `"купить [Бренд] [категория] Севастополь"` — есть ли Maxmobiles в топ-10? Кто конкурирует?
2. `"[Бренд] официальный магазин Россия [категория]"` — есть ли у бренда свой интернет-магазин?

При 2 вторичных брендах — 4 запроса параллельно.

**После получения результатов — для каждого бренда заполнить таблицу:**

```
## Мультибренд SERP: [Категория]

| Бренд | Офиц. магазин в топ-10 | Агрегаторов | Maxmobiles в топ-10 | Флаг |
|---|---|---|---|---|
| [Бренд] | да / нет → [домен] | N | да / нет | 🎯 / 👁 / — |
```

**Правила присвоения флага:**

| Условие | Флаг | Действие |
|---|---|---|
| Есть офиц. магазин бренда в топ-10 | **—** | Брендовый запрос нереалистичен. Бренд → только в description как ассортиментный сигнал. |
| Нет офиц. магазина + 0–1 агрегатор | **🎯** | Maxmobiles может конкурировать. Бренд → кандидат в title. |
| Нет офиц. магазина + 2–3 агрегатора | **👁** | Сложно, но мониторить. Бренд → description. |
| Нет офиц. магазина + 4+ агрегатора | **—** | Не включать в title. |

**Стратегические выводы Step 2e передать в Step 3:**
- Бренд с флагом **—** → упомянуть в description и H1 (как ассортиментный факт), **не включать в title**
- Бренд с флагом **🎯** → кандидат в title (Вариант с мультибрендом)
- Бренд с флагом **👁** → включить в description, в title опционально

### [v2] Step 2b-Wordstat — Частотный кластерный анализ (приоритетный метод)

**Выполнять всегда**, когда нет измеренной пользователем частоты по ключам (см. правило в Step 1). Использует MCP-инструменты `wordstat_top`, `wordstat_bulk`, `wordstat_regions`.

**Почему обязателен:** запрос, который сформулировал пользователь или который выглядит «правильным» по SERP-паттернам конкурентов, может оказаться не самым частотным в кластере. SERP-эвристика (Step 2b ниже) показывает, ЧТО пишут конкуренты в title — но не показывает, ЧТО реально ищут люди. Выбор ключа для H1/title должен опираться на измерение, а не на предположение.

**Алгоритм:**

1. **Seed-запрос** = формулировка из Step 1 (то, что дал пользователь или основной продукт/категория).
2. **Расширение кластера** — `wordstat_top(seed, num_phrases=30-40)` возвращает `totalCount` seed'а и associations (LSI/related-запросы). К результату добавить вручную обязательные варианты:
   - формальный термин ↔ разговорный синоним (например «внешний аккумулятор» ↔ «повербанк»)
   - **латиница ↔ кириллица** для любых буквенных обозначений в запросе (mAh↔мач, Pro↔про, GB↔гб и т.п.) — тестировать оба написания всегда, победитель непредсказуем в обе стороны (пример: «аккумулятор... мач» обгоняет «...mah» в 2 раза, а «повербанк... mah» обгоняет «...мач» — правило не одностороннее)
   - с брендом / без бренда — если категория мультибрендовая (Step 2e)
   - вариант без единицы измерения / модели (более широкий, для оценки масштаба ниши). **Обязательно прогнать через `wordstat_bulk` наравне с остальным кластером** — не оставлять как «справочную» запись без цифр. Если у страницы есть составной термин «бренд + линейка/модель» (например «Samsung Galaxy», «AirPods Pro») — протестировать также **урезанный вариант «[тип устройства] + голый бренд» без линейки** (например «смартфоны Samsung» отдельно от «смартфоны Samsung Galaxy» и отдельно от голого «Samsung»/«Самсунг» без «смартфоны»). Голый бренд без категорийного слова (просто «Samsung»/«Самсунг») почти всегда будет самым частотным в кластере, но проверить его тематическую релевантность через `wordstat_top(бренд)` → associations: если топ ассоциаций — другие товарные категории бренда (ТВ, бытовая техника, часы и т.п., не совпадающие с темой страницы), использовать только как справочный масштаб ниши, не как кандидата на H1/title.
3. **Региональная частота** — `wordstat_bulk(phrases=[кластер], regions=[977, 959])` (Крым + Севастополь по умолчанию для Maxmobiles — регионы можно расширить, если страница не гео-специфична). **Общероссийская частота из `wordstat_top` не используется для выбора H1/title** — только как справочный ориентир масштаба, так как магазин региональный и федеральный топ вводит в заблуждение (пример: «внешние аккумуляторы 10000 mah» — 1940 по России, но всего 60 в Крыму+Севастополе — почти в 33 раза меньше).
   - Число/падеж/род отдельно не тестировать — Yandex Wordstat нормализует словоформы автоматически и возвращает одинаковую частоту независимо от того, в каком виде введена фраза.
4. **Ранжирование** — отсортировать весь кластер по региональной частоте (убывание).

**Распределение по тегам:**

| Позиция в ранжированном кластере | Куда идёт |
|---|---|
| Топ-1 (максимальная региональная частота, формальный регистр, без brand-шума) | Основа H1 и title |
| 2–3 место (второй по частоте вариант / LSI-синоним / альтернативная латиница-кириллица) | Уточнение в title / первое предложение description |
| Остальной кластер (бренд-варианты, разговорные формы, длинный хвост) | Хвост description / FAQ-вопросы / семантическое ядро (Step 6) |

**[v2] Покрытие токенов у составных терминов — проверять перед тем, как выбирать «победителя» по частоте.** Когда в кластере есть термин-словосочетание и его подмножество-подстрока (например «Samsung Galaxy» и «Samsung»; «смартфоны Samsung Galaxy» и «смартфоны Samsung») — это НЕ всегда конкуренты за одно место. Поисковик матчит H1/title по отдельным леммам, а не только по точной фразе целиком: если более длинный термин содержит все слова короткого («Смартфоны Samsung Galaxy» содержит «Samsung» и «смартфоны» из «смартфоны Samsung»), он уже покрывает частотность короткого термина без потерь — сравнивать их «либо/либо» не нужно, длинный вариант просто добавляет дополнительный токен сверху. Реальный выбор нужен только когда термины действительно конкурируют за одно и то же место в теге (например два разных написания одной сущности — тогда работает правило script-separation ниже) или когда более короткий термин не является лемматическим подмножеством более длинного.

**[v2] Проверка каннибализации с дочерними подкатегориями — обязательна для хаб/родительских страниц, выполняется перед распределением по тегам выше.**

Если оптимизируемая страница — хаб/родительская категория, у которой в CS-Cart есть дочерние подкатегории со своими URL (например, `/vse-dlya-igr/` — родитель, `/vse-dlya-igr/pristavki/` — дочерняя), проверить через `site:домен [термин]` или прямой WebFetch дочерних URL: не занят ли термин из ранжированного кластера уже title/H1 одной из дочерних страниц.

Правило: термин, являющийся точным или почти точным заголовком дочерней подкатегории, **исключается из кандидатов на H1/title родителя** — независимо от его места в ранжировании по частоте. Хаб получает H1/title из следующего по частоте термина кластера, который не занят ни одной дочерней страницей (обычно это более широкий/собирательный термин — он и есть собственная, не пересекающаяся ниша хаб-страницы). Термины дочерних страниц не выбрасываются из работы совсем — они законно идут в хвост description хаба как перечисление подкатегорий (это не каннибализация, а нормальная внутренняя перелинковка).

Если неизвестно, оптимизирована ли дочерняя страница под этот термин — спросить пользователя, не предполагать самостоятельно.

**Утечка через паддинг короткого H1 — частая ошибка, проверять отдельно.** Если H1 короче 15 симв. и требует расширения (правило в Step 3), не добавлять занятый дочерней страницей термин в качестве слова-заполнителя — это возвращает каннибализацию через чёрный ход. Заполнять нейтральными словами из невостребованной части кластера или общими описательными словами, не пересекающимися с заголовками дочерних страниц.

Пример: у хаба «Всё для игр» термин-победитель кластера «игровые приставки» (1557) уже является title дочерней `/vse-dlya-igr/pristavki/` — исключён из title хаба. Термины «игровые ноутбуки» (4073) и «игровой ПК» (1768) — выше по частоте, но тоже соответствуют дочерним подкатегориям каталога — статус их оптимизации нужно уточнить у пользователя, прежде чем отдавать им H1/title хаба. Собственный, не занятый ни одной дочерней страницей термин хаба — «всё для игр» (1227).

**[v2] Каннибализация в description фильтр-страниц (цвет/память/аксессуар и т.п.) — отдельная проверка, даже когда H1/title конфликта не создают.**

Фильтр-страница (например, `/iphone/orange/` — цвет, `/iphone/256gb/` — память) обычно не конкурирует по H1/title с конкретной модельной страницей (`/iphone/iphone-17-pro/`), потому что целевые термины разные («айфон оранжевый» vs «iPhone 17 Pro»). Но если `description` **открывается** названием модели, у которой есть отдельная оптимизированная страница (проверить через `site:домен [модель]`, аналогично проверке дочерних подкатегорий выше) — это размывает тематический фокус фильтр-страницы: первая фраза должна сигнализировать «это про фильтр» (цвет/память/т.п.), а не «это про конкретную модель».

Правило: **первое предложение `description` и `WebPage.description` ведёт со значения фильтра** (цвет, объём памяти и т.п.), конкретные модели — только в поддерживающей части предложения или в хвосте. Модели упоминать можно и нужно (это легитимный контент, не каннибализация как таковая) — важен только порядок: фильтр первым, модель — support.

Пример: у страницы `/iphone/orange/` (цвет) описание **не должно** начинаться с «iPhone 17 Pro оранжевый в Maxmobiles» (это лид дочерней модельной страницы `/iphone/iphone-17-pro/»). Верно: «Оранжевый iPhone в Maxmobiles — iPhone 17 Pro и iPhone 17 Pro Max в цвете «Космический оранжевый»...» — фильтр (цвет) первым, модели во второй части.

**[v2] Приоритет частотности над визуальным стилем сайта — базовый принцип:** выбор написания (кириллица/латиница), термина (формальный/разговорный) и формулировки везде определяется измеренной региональной частотой, а не тем, как принято писать на остальном сайте, не «читабельностью» на глаз и не личным вкусом. Максимальный охват по SEO — приоритетнее визуальной консистентности бренда. Если топ-1 кластера — кириллица, использовать кириллицу, даже если весь остальной сайт пишет латиницей; если новый термин ломает привычный паттерн страниц — это ожидаемо, паттерн меняется вслед за данными, а не наоборот.

**Правило разделения H1/title по написанию единицы:** если кириллический и латинский варианты единицы измерения (мАч/mAh, про/Pro и т.п.) расходятся по частоте — H1 берёт топ-1 по частоте независимо от написания, принятого на сайте. title берёт следующий по частоте кластера термин (второе место), который по определению использует другое написание/регистр — так одновременно выполняется и максимальный охват, и правило «H1 не дублирует title» (Step 4). Если топ-1 и топ-2 совпадают по написанию — для title взять топ-3 или сформулировать вариант с коммерческим сигналом (Step 3, ось Б), но не жертвовать частотностью топ-1 в H1.

### Step 2b — Анализ ключевых слов (traffic scoring) — SERP-оценка, fallback если Wordstat недоступен

**Когда keyword data не предоставлена И Wordstat недоступен — оценка значимости по SERP:**

Если численные данные ключей отсутствуют (SERP-маршрут), оценить значимость по частоте появления ключа в title/H1 реальных конкурентов (не агрегаторов) из Step 2:

| 0 упоминаний | < 15 — пропустить |

Пометить в комментарии: «значимость оценена по SERP-частоте, не по keyword data».

Если реальных конкурентов ≤ 2, порог 3+ не применять. Ключ, присутствующий в title хотя бы одного реального конкурента, считать кандидатом в title (~40). Пометить: «мало реальных конкурентов — оценка по присутствию в title».

### Step 2c — Gap Analysis реальных конкурентов

Выполнять **всегда** (на обоих путях). Двухшаговый процесс: сначала из SERP-сниппетов, затем при необходимости WebFetch.

**Шаг 1 — быстрый анализ SERP-сниппетов:**
Для каждого реального конкурента из топ-10 извлечь прямо из результатов поиска:
- Title (виден в SERP напрямую)
- Наличие meta description: если сниппет содержит текст — есть; если пусто — отсутствует
- Кириллика в title/сниппете (да/нет)

**Шаг 2 — WebFetch: условно, максимум 2 параллельных запроса:**
Запускать только если в топ-10 найден ★-конкурент. Тогда параллельно:
- **1 WebFetch на URL страницы ★-конкурента** из SERP-результата
- **1 WebFetch на URL категории Maxmobiles** — для извлечения актуального ассортимента моделей

Если ★-конкурентов нет → 1 WebFetch на следующего реального конкурента из SERP.

**Fallback при сбое WebFetch:**
Если WebFetch вернул менее 200 символов читаемого контента — пропустить, перейти к следующему. Если все недоступны — использовать SERP-сниппеты; пометить: «данные из сниппета, не с полной страницы».

**Если реальных конкурентов не найдено (все топ-10 — агрегаторы):**

2. Записать: «Maxmobiles может стать первым специализированным Apple-магазином в топ-10 по данной нише»

### Step 3 — Генерация мета-тегов

**Цель не шаблон, а покрытие.** Максимально охватить высоковесовые ключи в рамках технических ограничений поисковых систем.

#### Правила по типу страницы

| Тип страницы | H1 — формат | Гео в H1 | «Купить» в title | Трейд-ин в USP |
|---|---|---|---|---|
| Топовая категория | `[Тип товара] Apple` или `[Тип товара] [Линейка]` | Нет | Внутри title | Да |
| Серия моделей | `[Тип товара] [Серия]` | Нет | Внутри title | Да |
| Конкретная модель | `[Тип товара] [Полное название модели]` | Нет | Первое слово в title | Да |
| Аксессуары | `[Тип аксессуара] Apple` или `[Тип аксессуара] [Линейка]` | Нет | Внутри title | Нет |
| Сервис / услуга | `[Услуга] в [Город]` | **Да — обязательно** | Не использовать | Нет |

**Специфика USP по типу:**
- **Аксессуары**: убрать трейд-ин и кредит, акцент на совместимости, доставке, гарантии
- **Сервис**: гарантия на работы, оригинальные запчасти Apple, срок ремонта, авторизованный сервисный центр Apple

#### H1 — что делает заголовок сильным

**Правила длины и формата:**
- Диапазон: **15–45 симв.** (проверять явно)
- Если название продукта < 15 симв. — добавить тип устройства перед ним: «Смартфон iPhone» (15 симв.)
- Запрещено: «Купить», «официальный», «авторизованный»

**Позитивные правила:**

1. **Порядок слов**: тип устройства → бренд/линейка → модель
   - ✅ «Смартфоны iPhone» — тип первым
   - ❌ «iPhone смартфоны» — обратный порядок хуже читается

2. **Полное официальное название** — не сокращать, не переводить на кириллицу:
   - ✅ «Смартфон iPhone 16 Pro Max»
   - ❌ «Смартфон Айфон 16 Про Макс»

3. **Для серий** — тип + серия:
   - ✅ «Смартфоны iPhone 16»
   - ❌ «iPhone 16» — без типа устройства

4. **Для мультибрендовых категорий** — основной бренд первым:
   - ✅ «Наушники Apple, Samsung»
   - ❌ «Samsung и Apple наушники»

5. **Для сервисных страниц** — услуга + устройство + город:
   - ✅ «Ремонт iPhone в Севастополе»
   - ❌ «Ремонт телефонов» — слишком широко

#### title — варианты по разным стратегическим осям

**Технические ограничения (жёсткие):**
- Google обрезает после ~65 симв. / 580px
- Yandex показывает до ~70 симв. без обрезки
- Нет keyword stuffing, нет caps, нет `!`, нет цен в ₽
- `Maxmobiles` всегда последним после `|`

**Три стратегические оси:**

| Ось | Что варьировать | Пример |
|---|---|---|
| **А — кириллика** | с/без кириллического варианта бренда | «iPhone» vs «iPhone (Айфон)» |
| **Б — коммерческий сигнал** | «купить» vs «цена» vs «гарантия» | «купить в Севастополе и Крыму» vs «цена и гарантия» |
| **В — гео** | двойное гео vs фолбэк | «в Севастополе и Крыму» vs «в Крыму» |

**Гео по умолчанию для товарных страниц — «в Севастополе и Крыму».**
Фолбэк на «в Крыму» применяется автоматически, если title с двойным гео > 65 симв.

**Порядок усечения (применять последовательно, пока title ≤ 65 симв.):**
1. Фолбэк гео: заменить «в Севастополе и Крыму» → «в Крыму» — экономит ~14 симв.
2. Убрать кириллику в скобках — экономит 8–15 симв.
3. Убрать коммерческий сигнал («купить»), оставить гео
4. Крайний случай: только продукт + Maxmobiles

**Правило первого слова в title:**
- Категории и серии: продукт первым (`iPhone 16 — купить в Севастополе и Крыму...`)
- Конкретная модель: «Купить» первым (`Купить iPhone 16 Pro Max в Севастополе и Крыму...`)
- Сервис: услуга первой (`Ремонт iPhone в Севастополе...`)

**Вывод вариантов:**

```
Вариант А: «[title]» ([N] симв.) — [ось]
→ Покрывает: [что именно]. Сильно для: [Яндекс / Google / оба].

Вариант Б: «[title]» ([N] симв.) — [ось]
→ Покрывает: [что именно]. Сильно для: [Яндекс / Google / оба].

Вариант В: «[title]» ([N] симв.) — [ось] (если есть)

→ Рекомендация: Вариант [X] — [одна фраза, обоснование из Step 2b данных].
```

#### meta description

**Технические ограничения:**
- Google Desktop: ~155 симв. видимый сниппет
- Яндекс расширенный сниппет: до 300+ симв.
- Оптимальная длина: **180–270 симв.**

**AEO-паттерн первого предложения:**

Структура: `[Продукт/Категория] в Maxmobiles — [прямой ответ на коммерческий запрос]`

AI-системы цитируют предложения, которые:
- Начинаются с объекта запроса (не с «Мы», не с «Maxmobiles»)
- Содержат прямой ответ в первых 10 словах
- Включают гео-сигнал (Крым / Севастополь)
- **[v2] Желательно, но не обязательно конкретное число** («с 2011 года», «1 год гарантии», «5.0, 1000+ отзывов») — обязательно число только в `WebPage.description` (см. JSON-LD раздел ниже), в обычном description оно подчиняется тому же Priority/demand-filter, что и остальные USP (см. ниже) — не вставляется автоматически в каждую страницу

❌ Слабо: «Maxmobiles предлагает iPhone с доставкой по России.»
✅ Сильно: «iPhone в Maxmobiles — гарантия 1 год, доставка СДЭК, самовывоз в Крыму с 2011 года.» (иллюстрирует структуру предложения; сами USP включены здесь при условии, что прошли demand-filter ниже — для iPhone обычно проходят, для нишевых категорий проверять отдельно)

**[v2] Фильтр по спросу (demand-filter) — обязателен перед вставкой любого факта в H1/title/description/WebPage.description:**

Бюджет символов в мета-тегах ограничен и конкурирует сам с собой: каждый символ, потраченный на факт без спроса, — это символ, не потраченный на ещё один синоним/LSI из Wordstat-кластера (Step 2b-Wordstat), который реально даёт охват. Компания уже описана на отдельных страницах сайта (Доставка, Гарантия, Trade-In) — дублировать эти детали в мета не нужно, если у дублирования нет собственного спроса или явного CTR-эффекта.

Перед вставкой факта — проверить:
1. Есть ли у него соответствие в Wordstat-кластере или семантическом ядре (реальный запрос/модификатор с частотой > 0)?
2. Если факта в кластере нет — это чисто информационный контент другой страницы сайта. Сократить до самой короткой правдивой формы или убрать вовсе, высвободив место под ещё один реальный запрос.

Пример (проверено Wordstat, регион Крым+Севастополь): «повербанк доставка сдэк» — 0, «повербанк гарантия» — 0, «аккумулятор точки самовывоза крым» — 0. Ни один составной запрос «товар+USP» не имеет измеримого спроса — такие фразы не увеличивают охват, только съедают бюджет. Правильно: «самовывоз в Крыму» (короткая правдивая форма, не «в 3 точках Крыма» — количество точек проверять негде, кроме страницы «Доставка»).

**Правило распространяется и на Priority 1 и 1b в таблице ниже** («гарантия 1 год», «доставка СДЭК», «самовывоз», «трейд-ин», «кредит») — формулировка «универсальный, всегда» там означает «всегда после демонд-валидации», а не освобождение от неё. Второй пример (регион Крым+Севастополь, категория «Всё для игр»): «игровые приставки доставка сдэк» — 0, «игровые приставки трейд-ин» — 0, «игровые приставки в кредит» — 0, «игровые приставки гарантия» — 0, «приставки самовывоз крым» — 0. Ни один Priority 1/1b USP по этой категории не прошёл фильтр — все исключены из description, бюджет отдан подкатегориям и кластерным терминам с измеренным спросом (например, «игровые ноутбуки», «шлем виртуальной реальности», «игры для PS5» — в отличие от USP-словосочетаний, у них частота > 0). Эти USP никуда не пропадают — они остаются в `.mm-trust-strip` и на отдельных страницах сайта (Доставка, Гарантия, Trade-In), где им и место; дублировать их в мета-бюджете без прироста охвата не нужно.

**Исключение — узкая, только для `WebPage.description`:** там число обязательно технически (см. JSON-LD раздел ниже, «Должен содержать минимум одно конкретное число») — это отдельное жёсткое требование формата, а не общее «включать USP-числа всегда». Для этого узкого случая, если ни один топически релевантный товарный номер не подходит, допустимо использовать «с 2011 года» как крайний случай — но это не отменяет demand-filter для обычного description выше и не освобождает Priority 3 от правила «по умолчанию не включать».

**[v2] Анти-дублирование терминов между полями:** H1, title и первое предложение description должны использовать РАЗНЫЕ термины из ранжированного Wordstat-кластера (позиции #1, #2, #3 соответственно — см. Step 2b-Wordstat), а не один и тот же топ-запрос три раза. Одинаковый термин во всех трёх полях не увеличивает охват (Yandex/Google это одна и та же лексема), а вариативность синонимов покрывает больше различных частотных кластеров одновременно.

**Структура description — фиксированный порядок:**

*Для товарных страниц:*
```
[AEO-первое предложение: 2-й термин кластера (Step 2b-Wordstat) + гарантия/доставка (если прошли demand-filter, см. Step 3) + CTA]  ← до ~155 симв.
[Ещё синонимы/LSI кластера + модельный ряд / топически релевантные характеристики продукта]  ← приоритет в хвосте, это основной объём
[E-E-A-T компании («с 2011 года», рейтинг) — ТОЛЬКО если место осталось]                     ← последний приоритет, по умолчанию не включать (см. таблицу приоритетов выше)
```

*Для сервисных страниц:*
```
[AEO-первое предложение: услуга + город + главный USP сервиса]      ← до ~155 симв.
[Перечень работ / услуг: что конкретно делается]                   ← приоритет в хвосте
[Условия сервиса: запчасти, срок, запись]                          ← после перечня работ
[E-E-A-T: авторизованный сервисный центр Apple, опыт с 2011 года]  ← в конце хвоста
```

**E-E-A-T в description — приоритет при нехватке места:**

**[v2] Компания vs категория — где живёт трастовый контент.** «С 2011 года», рейтинг 5.0/1000+ отзывов, экспертиза — это факты об МАГАЗИНЕ, не о конкретной категории товара. Им место в `LocalBusiness.description` (JSON-LD) — там они уже есть по правилам ниже — и в блоке доверия/бейдже отзывов на самой странице (`.mm-trust-strip`, rating-badge), которые не относятся к бюджету meta-тегов. Дублировать их ещё и в `description`/`WebPage.description` товарной страницы — трата бюджета на то, что и так видно в трёх местах, без прироста охвата и без нового доверия сверх уже показанного. **По умолчанию для товарных категорий эти факты в description/WebPage.description НЕ включать** — бюджет отдавать синонимам кластера (Step 2b-Wordstat) и характеристикам товара (заряды/Вт/порты и т.п. — тоже конкретные числа, но топически релевантные странице, а не магазину).

| Приоритет | Сигнал | Фраза | Условие |
|---|---|---|---|
| 1 — после демонд-валидации (универсальные, короткие) | Trustworthiness | «гарантия 1 год», «доставка СДЭК» | Пройти фильтр по спросу выше (composite-запрос «[топ-термин кластера] + USP» в Wordstat). Ненулевой спрос хотя бы по одной формулировке → включать первыми 155 симв. как CTR-триггер (подтверждено Gap Analysis — есть у всех живых конкурентов). 0 по всем формулировкам → не включать, это дублирует `.mm-trust-strip` и отдельные страницы сайта без прироста охвата |
| 1b — условно (см. [v2] тематическая релевантность ниже) | Trustworthiness | «трейд-ин», «кредит» | Только если категория/тип страницы это поддерживает — НЕ подставлять автоматически. Плюс демонд-валидация как в Priority 1 |
| 2 — обязательно (сервис) / условно (Apple-категории) | Expertise | «авторизованный сервисный центр Apple» (сервис — топически релевантно услуге) / «старейший Apple-эксперт в Крыму» (только Apple-категории, где это тоже по теме страницы) | Для мультибрендовых/не-Apple товарных категорий — не включать вовсе, не заменять нейтральной фразой, если место можно отдать синониму кластера |
| 3 — последний приоритет, только если не осталось реальных синонимов/LSI и характеристик товара | Experience/Authoritativeness | «с 2011 года», рейтинг 5.0/1000+ отзывов | По умолчанию НЕ включать в товарные description — уже есть в `LocalBusiness.description` и на странице (`.mm-trust-strip`, rating-badge). Использовать только как заполнитель, если после Priority 1/1b/2 и всех реальных ключей кластера в бюджете 180–270 симв. ещё осталось место |

**[v2] Проверка тематической релевантности USP — обязательна перед выводом description.**

Не все Approved USPs подходят каждой категории — «обязательно» в таблице выше НЕ отменяет правило по типу страницы (таблица «Специфика USP по типу» в начале Step 3). При конфликте между «обязательно всегда» и правилом типа страницы — **правило типа страницы побеждает**.

| USP | Включать, если | Убрать / заменить, если |
|---|---|---|
| Трейд-ин | Категория — Apple-устройства (iPhone/MacBook/iPad/Watch) или дорогой аксессуар (>10 000 ₽), где скидка от Trade-In заметна покупателю | Дешёвые аксессуары (повербанки, кабели, чехлы, зарядки, <5 000 ₽) — вычет по Trade-In на них незаметен и не соответствует интенту покупателя аксессуара |
| Кредит | Покупки, где рассрочка реально имеет смысл (смартфоны, ноутбуки, планшеты, дорогая техника) | Дешёвые аксессуары <5 000 ₽ — кредит на них не ищут и не оформляют |
| «Старейший Apple-эксперт в Крыму» | Категория — Apple-бренд (iPhone/Mac/iPad/Watch/AirPods) или Apple-сервис | Мультибрендовые / не-Apple категории (повербанки, аксессуары Xiaomi/Anker/UGREEN/VLP и т.п.) — заменить на нейтральную формулировку: «магазин электроники в Крыму с 2011 года» / «работаем в Крыму с 2011 года». Число «2011» сохранить всегда — это факт о компании, а не об Apple-специализации |

**Approved USPs** (список допустимых формулировок — условия включения см. в таблицах Priority 1/1b/2/3 и тематической релевантности выше; «универсальный» здесь означает «доступен для любой категории при выполнении условия», не «включать без проверки»):
- «гарантия 1 год» — универсальный, Priority 1, после demand-filter
- «доставка СДЭК» / «доставка по России через СДЭК» — универсальный, Priority 1, после demand-filter
- «самовывоз в Крыму» / «самовывоз в Севастополе, Симферополе, Ялте» — универсальный, Priority 1, после demand-filter
- «трейд-ин» / «Trade-In» — условный, Priority 1b: тематическая релевантность + demand-filter
- «кредит» / «покупка в кредит» — условный, Priority 1b: тематическая релевантность + demand-filter
- «авторизованный сервисный центр Apple» — только сервисные страницы, Priority 2
- «старейший Apple-эксперт в Крыму» — только Apple-категории, Priority 2, см. таблицу тематической релевантности
- «магазин электроники в Крыму с 2011 года» / «работаем в Крыму с 2011 года» — нейтральный аналог для мультибрендовых/не-Apple категорий, те же условия, что у строки выше
- «с 2011 года» / «более 15 лет» — Priority 3, последний приоритет — по умолчанию НЕ включать в товарный description (см. таблицу Priority выше), только как заполнитель при остатке места
- «бонусная программа» — универсальный, но также подчиняется фильтру по спросу (Step 3) как и остальные USP

> Запрещено везде: «рассрочка», эмодзи, цены в ₽, `!`
> «официальный» и «авторизованный» — запрещены в H1 и title. В description «авторизованный сервисный центр» допустимо как фактический USP.

**Многобрендовые категории:**
- H1: `[Тип товара] Apple, [Бренд2]` — не более 2 брендов
- title: `[Тип товара] [Бренды] — купить в Крыму | Maxmobiles`
- Остальные бренды → description

#### [v2] JSON-LD description — строгие правила для нейроответов

**`WebPage.description`** — главный источник для Яндекс Нейро и ChatGPT Search.

- Должен содержать **полное первое предложение** meta description (не обрезать на 155 симв.)
- **[v2] НЕ начинать** с «Maxmobiles», «Мы» — начинать с объекта запроса
- **[v2] Должен содержать минимум одно конкретное число.** Приоритет — топически релевантное число о товаре/категории («1 год гарантии», «заряжает 3–4 раза», «22,5 Вт», «3 порта»), а не о компании. Числа о компании («с 2011 года», «5.0, 1000+ отзывов») — только если больше нечем закрыть требование числа: они уже есть в `LocalBusiness.description` и на странице (`.mm-trust-strip`, rating-badge), дублировать их здесь по умолчанию не нужно (см. фильтр по спросу и таблицу приоритетов E-E-A-T выше)
- Длина: **120–180 симв.**

✅ Правильно (мультибренд/аксессуар — число о товаре): «Аккумулятор 20000 mAh (повербанк) в Maxmobiles — гарантия 1 год, доставка СДЭК, самовывоз в Крыму. Заряжает смартфон 3–4 раза.»
✅ Допустимо (Apple-категория, число о компании как крайний случай, если товарного числа нет): «iPhone в Maxmobiles — гарантия 1 год, доставка СДЭК, самовывоз в Крыму с 2011 года.»
❌ Неправильно: «Maxmobiles — Apple-магазин в Крыму. Здесь вы найдёте широкий выбор iPhone.»

**`LocalBusiness.description`** — 2–3 предложения: специализация + гео + главный USP. Не повторять `WebPage.description` дословно.

Пример: «Maxmobiles — старейший Apple-эксперт в Крыму с 2011 года. Самовывоз в Севастополе, Симферополе, Ялте. Доставка СДЭК по всей России, Trade-In, авторизованный сервисный центр Apple.»

**[v2] `LocalBusiness.sameAs` — обязательно:**

```json
"sameAs": [
  "https://yandex.ru/maps/org/237809440282"
]
```

### Step 6 — Семантическое ядро (запросы для отслеживания)

**Выводить по запросу.** После Step 5 добавить одну строку:
> Нужно семантическое ядро для Яндекс Вебмастера / GSC?

**Принцип:** только запросы, подтверждённые SERP-анализом или keyword data.

**Формат:** нижний регистр, без знаков препинания. Пример: `купить iphone 16 pro в крыму`.

**Объём:** ориентир **15–25 запросов** (Tier 0 + Tier 1 + Tier 2 + Tier 3 суммарно) — но это потолок, не обязательный план. Если после валидации через Wordstat (см. ниже) реальных запросов меньше — выводить меньше. Не добивать список до 15–25 непроверенными шаблонными комбинациями ради количества: 10 подтверждённых запросов лучше 20, из которых половина — нулевая по факту.

**[v2] Для нишевых/недорогих категорий (аксессуары, спецификация/ёмкость и т.п.) шаблон «[продукт] + город» может не подтверждаться спросом вообще** — проверено на «повербанк 20000 mah крым/севастополь», «внешний аккумулятор 20000 мач крым» — все дали 0. В этом случае реальное семантическое ядро строить из самого измеренного Wordstat-кластера (Step 2b-Wordstat) — там уже есть настоящие частоты, а не из шаблонных гео-комбинаций Tier 1 ниже. Шаблоны Tier 1–3 ниже — рабочая гипотеза для категорий с более высоким чеком (смартфоны, ноутбуки), где geo-комбинированный поиск более вероятен; для дешёвых акссессуаров — обязательно проверять, не подставлять вслепую.

**[v2] Если выполнялся Step 2b-Wordstat** — сортировать запросы внутри каждого Tier по измеренной региональной частоте (убывание), не только по флагу агрегаторов. Флаг 🎯/👁 определяет Tier (конкурентность), частота Wordstat определяет порядок внутри Tier (приоритет).

**[v2] Валидация составных USP-запросов перед выводом Tier-списка — обязательна.** Шаблонные строки вида «[продукт] трейд-ин крым», «[продукт] доставка сдэк», «[продукт] с гарантией купить», «[продукт] в кредит [город]» — это гипотезы по шаблону, не факты. Прогнать такие составные запросы через `wordstat_bulk` перед финальным выводом; если частота 0 (частый случай — люди не ищут «товар + условие покупки» единой фразой), **исключить их из Tier-списка целиком**, не понижать в приоритете. Не отправлять непроверенные шаблонные запросы в PixelPlus (Step 6a) — там должны быть только запросы с измеренной частотой > 0.

Это та же проверка, что уже выполнялась в Step 3 (demand-filter для Priority 1/1b USP в description) — если composite-запросы «[топ-термин] + USP» там уже прогонялись через `wordstat_bulk`, повторно их не гонять, использовать готовый результат.

**Нежелательные интенты — НЕ включать:**

```
❌ Информационные:   «[продукт] обзор», «[продукт] отзывы», «[продукт] характеристики»
❌ Сервисные (для товарных страниц): «[продукт] ремонт», «[продукт] прошивка»
❌ Конкурентные бренды: запросы с упоминанием конкурентов
❌ Недостижимые: «[продукт]» без гео при 4+ агрегаторах в топ-10
```
